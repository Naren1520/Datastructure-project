Data Structures Used in NexStock

NexStock uses a combination of basic and efficient data structures designed for educational purposes while maintaining practical functionality.

Primary Structures

1. Product Struct

struct Product { int id; // 4 bytes char name[50]; // 50 bytes float price; // 4 bytes int quantity; // 4 bytes }; // Total: 62 bytes

Purpose: Store individual product information

Fields: ID, Name, Price, Quantity

2. Rental Struct

struct Rental { long rentalId; // 8 bytes int productId; // 4 bytes char productName[50]; // 50 bytes char renterName[50]; // 50 bytes char rentDate[20]; // 20 bytes char returnDate[20]; // 20 bytes char phoneNumber[15]; // 15 bytes char address[100]; // 100 bytes float amountPaid; // 4 bytes char status[20]; // 20 bytes }; // Total: 361 bytes

Purpose: Store rental transaction details

Fields: 10 fields for complete rental tracking

Array Data Structures

inventory[] Array:
struct Product inventory[MAX]; // MAX = 100
  • Type: Fixed-size static array

rentals[] Array:
struct Rental rentals[MAX]; // MAX = 100
  • Type: Fixed-size static array
  • Capacity: 100 rental records
  • Memory: 36.1 KB (100 × 361 bytes)

  • Use: Store all rental transactions

Algorithms & Techniques

NexStock implements various classic algorithms for data searching and sorting.

Sorting Algorithms - Bubble Sort

All three sorting operations use the Bubble Sort algorithm, which is ideal for educational purposes.

Algorithm Characteristics:
  • Compares adjacent elements repeatedly
  • Swaps them if they are in wrong order
  • Multiple passes until array is sorted
  • Simple to understand and implement
  • Not optimal for large datasets
Sort by ID
Integer comparison based sorting
Time: O(n²)
Space: O(1)
Sort by Name
String comparison using strcmp()
Time: O(n²)
Space: O(1)
Sort by Price
Floating-point value comparison
Time: O(n²)
Space: O(1)

Search Algorithm - Linear Search

Search Product by ID
Sequential scan through entire array
Time: O(n)
Space: O(1)
Algorithm Details:
for (int i = 0; i < count; i++) { if (inventory[i].id == searchId) { // Found - return product return; } } // Not found
Characteristics:
  • Checks each element sequentially
  • Best case: O(1) - found at first position
  • Worst case: O(n) - found at last or not found
  • No pre-sorting required
  • Works on any data

Operations Complexity Analysis

System Architecture

NexStock follows a modern 3-tier architecture with educational visualizations.

System Architecture Layers

Layer 1: Frontend (HTML/CSS/JavaScript)
  • User interface for inventory operations
  • Interactive forms and real-time feedback
  • Visual flow diagrams
  • Code viewer with highlighting
Layer 2: HTTP/REST API
  • JSON request/response format
  • Express.js routing
  • RESTful endpoint design
  • POST/GET/PATCH/DELETE operations
Layer 3: Backend (Node.js Server)
  • Request parsing and validation
  • C backend integration
  • Response formatting
Layer 4: C Backend Logic
  • Core data processing algorithms
  • Product and rental management
  • Sorting and searching operations
  • File I/O operations
Layer 5: Storage (Files & JSON)
  • Persistent data storage
  • inventory.json - Product data
  • rentals.json - Rental data

Data Flow Example: Adding a Product

1. Frontend: User fills form and clicks "Add Product" ↓ 2. HTTP: POST request sent to /api/c/product/add with JSON data ↓ 3. Server (Node.js): Receives request, parses JSON ↓ 4. Routing: Routes to addProduct handler ↓ 5. C Backend: - Validates product ID (check for duplicates) - Checks inventory capacity - Creates Product struct - Adds to inventory array - Increments counter ↓ 6. Storage: Saves updated inventory to inventory.json ↓ 7. HTTP Response: {success: true, message: "Product added"} ↓ 8. Frontend: Updates UI, shows success message ↓ 9. Code Viewer: Highlights executed C code lines
